## 1. 이슈 개요 (Issue Overview)
기존 SmartBox 펌웨어는 InfluxDB 원격 데이터 전송(`TelemetryManager`) 및 자동 펌웨어 업데이트 검사(`AutoOtaManager`) 동작 시 수반되는 HTTP/HTTPS 동기(Blocking) 네트워크 호출이 메인 `loop()` 스레드 내에서 직간접적으로 처리되어 왔습니다. 이로 인해 Wi-Fi 환경의 일시적 지연이나 네트워크 시간 동기화(NTP) 확인 시 메인 루프가 초 단위로 지연(Freezing)되는 현상이 발생하여, 모터와 거리 감지 루프의 실시간 제어가 마비될 위험이 있었습니다.

## 2. 원인 분석 (Root Cause Analysis)
- **메인 스레드 대기(Blocking):** 주기적으로 호출되는 InfluxDB API 및 OTA 버전 체크가 비록 백그라운드 태스크를 별도 생성했더라도 상태 검사나 NTP 검증 등으로 인해 메인 루프 상에서 일시 지연을 유발할 수 있습니다.
- **스레드 경합(Data Race):** 데이터를 백그라운드 스레드에서 전송하도록 스레드를 온전히 격리시킬 경우, 메인 루프가 센서 및 제어 상태를 기록하는 멤버와 백그라운드 스레드가 이를 읽어가는 시점 간의 스레드 충돌 및 데이터 손상 가능성(Data Race)이 존재합니다.
- **안전 제어 유실:** 백그라운드에서 다운로드 후 바로 플래싱으로 진입할 경우, 모터가 현재 기동 중인지 검증하지 않아 물리적 파손이나 릴레이 합선 사고로 이어질 위험이 있습니다.

### 3.1. 전용 백그라운드 태스크 (`NetworkTask`) 구축
메인 루프에서 실행되던 네트워크 갱신 루틴을 완전히 전용 FreeRTOS 태스크로 분리하여 백그라운드에서 실행하도록 이관하였습니다.
- [main.cpp](../src/main.cpp)에 persistent 백그라운드 루프 `NetworkTask`를 정의하여 16KB의 넉넉한 스택 공간을 주었습니다.
- `setup()`에서 `xTaskCreate`로 낮은 우선순위(Priority 1)의 백그라운드 태스크로 `NetworkTask`를 구동하고, 메인 `loop()`에서는 `TelemetryManager::update()` 및 `AutoOtaManager::update()` 호출을 전면 제거하여 FSM 루프에 미치는 지연 시간을 0ms로 단축시켰습니다.

### 3.2. C++ standard `std::recursive_mutex`를 이용한 Thread-Safety 보장
스레드 격리에 따른 데이터 레이스(Data Race)를 해결하기 위해 중첩 잠금이 가능한 `std::recursive_mutex` 구조를 도입하였습니다.
- [SmartBoxController.h](../src/SmartBoxController.h)에 `mutable std::recursive_mutex dataMutex;`를 추가하였습니다.
- [SmartBoxController.cpp](../src/SmartBoxController.cpp)의 핵심 데이터 쓰기 영역(`readSensors()`, `updateDistanceBuffer()`, `transitionTo()`) 및 거리와 전압, 전류, FSM 상태를 외부에 노출하는 모든 getter 메소드에 `std::lock_guard<std::recursive_mutex>`를 도입하였습니다.
- `std::recursive_mutex`를 활용함으로써 `canSendTelemetry()` 내부에서 다시 `isMotorRunning()`을 호출하는 중첩 락 상황에서도 데드락(Deadlock)이 발생하는 현상을 예방하였습니다. 또한 PC 기반 유닛 테스트 환경에서도 컴파일 가드 없이 네이티브 컴파일을 지원합니다.

### 3.3. 동적 태스크 오버헤드 제거 (정적 태스크 이관)
- 기존에 네트워크 시도 시마다 동적으로 `xTaskCreate`를 호출하여 백그라운드 서브 스레드를 생성 및 제거하던 로직을 모두 제거하였습니다.
- InfluxDB 포인트 제출 및 OTA 업데이트가 상시 대기 중인 `NetworkTask` 스레드 내부에서 동기적(Synchronous)으로 수행되어, 힙 파편화 및 FreeRTOS 스케줄러 오버헤드를 대폭 절감하였습니다.

### 3.4. 🚨 Pre-OTA Safety Interlock 강화
- 자동 및 수동(웹 대시보드 API) OTA 다운로드 및 플래싱 진입 전에 즉시 FSM을 `STATE_OTA_UPDATING`으로 천이하고 모든 릴레이를 물리적으로 차단(`forceAllRelaysOff()`)합니다.
- 실제 플래싱 쓰기를 진행하는 `httpUpdate`가 구동되기 전, 모터의 기동 전류를 읽어 전류값이 완전히 차단(50mA 미만)되고 안전이 확보된 후에만 펌웨어 업데이트를 수행하도록 동기화 대기 검증 루프를 추가하였습니다.
