# SmartBox 새벽 4시 연결 끊김 현상 분석 리포트

> **작성일:** 2026-07-20  
> **분석 대상 브랜치:** main (`dbcfff6`)  
> **이슈 현상:** 매일 새벽 4시경 Home Assistant에서 SmartBox 기기가 오프라인으로 표시되었다가 수십 초 후 자동 복구됨

---

## 1. 결론 (Summary)

> [!IMPORTANT]
> 새벽 4시 연결 끊김은 **`PowerManager`의 의도된 자동 일일 재부팅 동작**으로 판단됩니다. 이는 비정상적인 오류가 아니라 소프트웨어 설계에 의해 예정된 동작입니다. 단, 재부팅 후 MQTT 재연결 과정에서 LWT(Last Will and Testament) 메시지가 발행되어 Home Assistant가 기기를 오프라인으로 표시하는 것이 관찰됩니다.

---

## 2. 근본 원인 분석 (Root Cause Analysis)

### 2.1. PowerManager 자동 재부팅 코드 분석

[PowerManager.cpp](../src/PowerManager.cpp)의 `update()` 함수 내 재부팅 트리거 로직:

```cpp
// PowerManager.cpp:39-57
int currentHour = timeinfo.tm_hour;

// Proactive daily reboot scheduling at 04:00 AM
// Only reboots when FSM is in IDLE and motor is stopped to ensure safety
if (currentHour == 4 && controllerPtr->getCurrentState() == STATE_IDLE 
    && !controllerPtr->isMotorRunning()) {
    Preferences prefs;
    if (prefs.begin("smartbox", false)) {
        int lastRebootDay = prefs.getInt("reboot_day", -1);
        if (lastRebootDay != timeinfo.tm_yday) {
            prefs.putInt("reboot_day", timeinfo.tm_yday);
            prefs.end();
            Serial.printf("[POWER] Proactive scheduled reboot (04:00 AM, day of year: %d). Restarting...\n", timeinfo.tm_yday);
            delay(500);
            ESP.restart();
        }
        prefs.end();
    }
}
```

**분석:**

| 항목 | 내용 |
| :--- | :--- |
| **트리거 조건** | `currentHour == 4` AND `STATE_IDLE` AND 모터 정지 |
| **중복 방지** | NVS `reboot_day`에 실행 날짜 기록, 하루 1회 보장 |
| **폴링 주기** | `update()` 15초 간격 → 04:00:00~04:14:45 사이 첫 IDLE 시점에 실행 |
| **안전 조건** | 뚜껑 동작 중(OPENING/CLOSING)이면 재부팅 지연 (IDLE 대기) |

---

### 2.2. 재부팅 이후 연결 복구 타임라인

`ESP.restart()` 호출 이후 시스템 복구 흐름:

```
T+0s    ESP.restart() 호출
T+0~3s  ESP32-C6 부트로더 및 펌웨어 초기화
T+3s    setup() 실행 시작
          - hardware.init()
          - ConfigManager::loadConfig()
          - SmartBoxController::init()
          - WifiManager::init() -> SoftAP 활성화 + STA 연결 시작
T+5~15s Wi-Fi STA 연결 완료 (공유기 응답 시간에 따라 상이)
T+15s   ARDUINO_EVENT_WIFI_STA_GOT_IP 이벤트
          - configTime() 호출 -> NTP 동기화 시작
T+16s   NetworkTask 생성 및 시작
          - AutoOtaManager::update() 최초 OTA 버전 체크 시작
          - MqttManager::update() MQTT 연결 시도
T+16~30s MQTT TLS 핸드셰이크 (1~5초 소요)
T+30s~  MQTT 연결 성공
          - smartbox/status {"status":"online"} 발행 (retain=true)
          - HA Auto Discovery 설정 패킷 16개 발행
          - 현재 FSM 상태 publishEventState() 강제 발행

총 오프라인 예상 시간: 15~30초 (네트워크 환경에 따라 최대 60초)
```

---

### 2.3. Home Assistant에서 오프라인으로 표시되는 메커니즘

MqttManager의 LWT(Last Will and Testament) 설정:

```cpp
// MqttManager.cpp:39-40
const char* willPayload = "{\"status\":\"offline\",\"device_id\":\"ESP32C6_SMARTBOX_01\"}";
m_mqttClient.setWill("smartbox/status", 1, true, willPayload);
```

**동작 원리:**

1. ESP32가 재부팅되면 MQTT 연결이 갑자기 끊어집니다.
2. MQTT 브로커는 Keep-Alive 타임아웃(60초 설정) 이후 LWT 메시지를 자동 발행합니다.
3. Home Assistant는 `smartbox/status`에서 `{"status":"offline"}` 수신 시 기기를 오프라인으로 표시합니다.
4. ESP32 재부팅 완료 후 `{"status":"online"}`을 발행하면 온라인으로 복구됩니다.

> [!NOTE]
> LWT 발행 지연(Keep-Alive 60초)으로 인해 실제 재부팅 후에도 최대 60초간 Home Assistant가 오프라인을 감지하지 못하다가 이후 오프라인 → 온라인 전환이 나타날 수 있습니다.

## 3. 비정상 연결 끊김 가능성 분석

만약 새벽 4시에 연결이 끊겼다가 복구되는 시간이 **5분 이상** 소요되거나 **복구되지 않는** 경우, 다음 원인을 추가 점검해야 합니다.

### 3.1. AutoOtaManager와의 타이밍 충돌

`AutoOtaManager`가 1시간 주기로 OTA 버전을 체크합니다. 새벽 4시 재부팅 이후 즉시 OTA 첫 부팅 체크가 실행되고, 새 버전이 있으면 `STATE_OTA_UPDATING`으로 전이되어 업데이트를 진행합니다.

```cpp
// AutoOtaManager.cpp:69-80
bool firstRun = (lastOtaRunTime == 0);
if (firstRun || intervalElapsed) {
    lastOtaRunTime = now;
    if (firstRun) {
        Serial.println("[AUTO-OTA] First boot: running initial OTA version check...");
    }
    runOtaProcess(false);
}
```

OTA 업데이트가 실행되면 MQTT 연결이 추가 시간 동안 끊어진 상태로 유지될 수 있습니다.

**추정 추가 소요 시간:**
* NAS 버전 체크: ~2~5초
* 펌웨어 다운로드 및 플래시: ~60~120초
* 자동 재부팅 후 재연결: ~30초

**총 연결 끊김 예상: 최대 3~5분** (OTA 업데이트가 있는 경우)

### 3.2. Wi-Fi 환경 열화 (새벽 특유의 간섭)

새벽 4시는 ISP 라우터의 야간 자동 재시작 또는 DHCP 갱신 구간과 겹칠 수 있습니다.

`WifiManager::update()`의 재연결 로직:
```cpp
// WifiManager.cpp:290-295
if (!_staSsid.isEmpty() && !connected && (millis() - lastConnectRetry >= 15000)) {
    lastConnectRetry = millis();
    Serial.printf("[WIFI] Attempting connection retry to AP '%s'...\n", _staSsid.c_str());
    WiFi.disconnect();
    WiFi.begin(_staSsid.c_str(), _staPass.c_str());
}
```

Wi-Fi 재연결 재시도 주기가 15초이므로, 공유기가 응답하지 않는 동안 최대 여러 회의 재시도가 발생합니다.

### 3.3. NTP 동기화 미완료

`PowerManager`는 NTP 동기화가 완료된 경우에만 `getLocalTime()`이 성공합니다.

```cpp
// PowerManager.cpp:33-37
struct tm timeinfo;
if (!getLocalTime(&timeinfo, 0)) {
    // NTP time not synchronized yet
    return;
}
```

재부팅 직후 NTP 동기화가 완료되기 전에는 PowerManager가 자동 재부팅을 실행하지 않습니다. NTP 서버 응답 지연 시 재부팅 타이밍이 04:00 이후로 밀릴 수 있습니다.

## 4. 연결 끊김 유형별 판별 기준

| 끊김 시간 | 가장 유력한 원인 |
| :--- | :--- |
| **15~30초** | 정상 자동 재부팅 (PowerManager) |
| **30~60초** | 정상 재부팅 + Wi-Fi 연결 지연 또는 MQTT TLS 핸드셰이크 재시도 |
| **60초~5분** | 자동 재부팅 + 초기 OTA 버전 체크 실행 (신규 버전 없는 경우) |
| **3~7분** | 자동 재부팅 + OTA 업데이트 완료 및 재부팅 |
| **5분 이상 지속** | Wi-Fi 환경 이상, MQTT 브로커 문제, 또는 예외 상황 |

## 5. 진단 방법

### 5.1. Serial 로그 확인

PlatformIO 시리얼 모니터에서 04:00경 아래 로그를 확인하면 자동 재부팅을 확인할 수 있습니다:

```
[POWER] Proactive scheduled reboot (04:00 AM, day of year: NNN). Restarting...
[SYSTEM] Starting SmartBox Controller...
[SYSTEM] Initialization finished. FSM Loop running.
[WIFI] On-Demand SoftAP 'SmartBox-WiFi' active for 5 mins.
[WIFI] Station got IP address: 192.168.x.x
[TIME] NTP Sync initialized (KST, UTC+9).
[MQTT] Connecting to Secure MQTT broker (Async)...
[MQTT] Connected to secure broker successfully!
```

### 5.2. MQTT 로그 토픽 구독

Home Assistant MQTT 도구 또는 MQTT 클라이언트에서 다음 토픽을 구독하여 실시간 확인:

```
smartbox/status       → offline -> online 전환 시각 확인
smartbox/log          → WARN/ERROR 수준 원격 로그
smartbox/event/state  → FSM 상태 전이
```

### 5.3. Home Assistant 로그에서 확인

Home Assistant의 **[개발자 도구] > [이벤트]** 또는 **[로그북]** 에서 `SmartBox` 기기의 오프라인/온라인 전환 시각을 기록하여 패턴을 분석하십시오.

## 6. 권장 대응 방안

### 6.1. 현재 동작 유지 (권장)

자동 재부팅은 메모리 누수 방지 및 장기 안정성을 위해 설계된 기능입니다. 현재 동작(15~30초 오프라인 후 자동 복구)은 정상 범위이며 유지를 권장합니다.

### 6.2. 재부팅 시각 변경

트래픽이 없는 시간대로 변경하려면 Home Assistant에서 `config_ota_hour` Select 엔티티로 자동 OTA/재부팅 시각을 조정하거나, MQTT로 직접 명령:

```json
{"command": "set_config", "key": "otaHour", "value": 3}
```

> [!NOTE]
> PowerManager의 재부팅 시각(04:00 AM)과 AutoOtaManager의 OTA 시각(`config.otaHour`)은 별도로 관리됩니다. PowerManager 재부팅 시각을 변경하려면 [PowerManager.cpp](../src/PowerManager.cpp)의 `currentHour == 4` 조건을 코드로 수정해야 합니다.

### 6.3. MQTT Keep-Alive 최적화

현재 Keep-Alive가 60초로 설정되어 있습니다. LWT 발행 지연을 줄이려면 Keep-Alive 값을 낮추는 것을 고려할 수 있으나, 네트워크 트래픽이 증가합니다.

```cpp
// MqttManager.cpp:36
m_mqttClient.setKeepAlive(60); // 현재 60초, 필요시 30초로 단축 가능
```
