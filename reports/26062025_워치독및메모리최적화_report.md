# 📄 보고서: 장기 생존성 확보를 위한 FreeRTOS Task Watchdog 및 힙 메모리 최적화 구현

- **작성일자:** 2026년 06월 20일
- **주제:** ESP32-C6 SmartBox의 장기 무인 구동 안정성을 위한 FreeRTOS Task Watchdog Timer(TWDT) 구축, 동적 String 예약(reserve)을 통한 힙 단편화 방지, 새벽 시간대 선제적 자동 재부팅(Proactive Reboot) 구현

---

## 1. 이슈 개요 (Issue Overview)
야외에서 장기간 무인으로 동작해야 하는 스마트 쓰레기통 제어기(SmartBox)의 특성상 시스템의 영구적인 고착(Hang)을 방지하고 장기 작동 안정성을 확보해야 합니다. 
1. **힙 메모리 단편화 위험:** 기존 `WebDashboard.cpp`와 `WifiManager.cpp` 등에서 주기적으로 JSON 응답을 구성할 때 dynamic `String` 객체를 반복적으로 사용하여 힙(Heap) 메모리 단편화로 인한 메모리 부족(OOM) 오류가 발생할 가능성이 높았습니다.
2. **소프트웨어 고착(Hang) 무방비:** 하드웨어 노이즈, 네트워크 소켓 대기 지연 등으로 인해 메인 루프나 백그라운드 태스크가 멈추었을 때 자동으로 복구하는 안전장치(Watchdog Timer)가 없었습니다.
3. **자동 펌웨어 업데이트(OTA) 연동 문제:** 워치독 타이머 적용 시, 비교적 장시간 블로킹되는 펌웨어 플래싱 과정(OTA)에서 워치독이 리셋을 유발하여 시스템이 벽돌(Brick)이 될 위험이 있었습니다.

---

## 2. 원인 분석 (Root Cause Analysis)
- **동적 `String` 할당:** C++ `std::string` 또는 Arduino `String` 클래스는 동적 추가(`+=`) 시 수시로 메모리를 재할당합니다. 이 과정에서 힙 영역에 다양한 크기의 빈 구멍이 생겨 큰 단일 메모리 블록을 할당하지 못해 런타임에 기기가 오동작할 위험이 있습니다.
- **ESP-IDF v5의 TWDT API 변화:** ESP32 Arduino Core 3.0(ESP-IDF 5.x) 환경에서는 기존의 `esp_task_wdt_init(timeout, panic)` 함수 사용 시 시그니처 매칭 문제로 컴파일 에러가 발생합니다. 따라서 새로운 `esp_task_wdt_config_t` 구조체를 활용한 설정 방식으로 구현해야 합니다.

---

## 3. 해결 설계 및 구현 (Solution Design & Implementation)

### 3.1. 힙 단편화 방지를 위한 String reserve 적용
- **[WebDashboard.cpp](../src/WebDashboard.cpp)**: 
  - `/api/status` 핸들러 내부에서 JSON 빌드를 시작하기 전 `json.reserve(512);`를 호출하여 한 번에 버퍼를 메모리에 확보함으로써 빈번한 메모리 할당을 억제했습니다.
  - `/api/wifi/scan` 핸들러에서도 응답 시 `json.reserve(results.length() + 64);`를 적용하여 문자열 접합 시 발생하는 동적 재할당을 방지했습니다.
- **[WifiManager.cpp](../src/WifiManager.cpp)**:
  - `getScanResultsJson()` 함수 내에서 스캔된 Wi-Fi 네트워크 수 `n`에 기반하여 `json.reserve(32 + n * 64);`를 호출해 힙 단편화를 근본적으로 차단했습니다.

### 3.2. FreeRTOS Task Watchdog Timer (TWDT) 통합 및 OTA 예외 처리
- **[main.cpp](../src/main.cpp)**:
  - ESP32-C6 아두이노 코어 버전에 독립적이도록 `ESP_ARDUINO_VERSION_MAJOR >= 3` 매크로 분기를 통해 새 `esp_task_wdt_config_t` 구조체와 기존 `esp_task_wdt_init` 함수 호출 형식을 모두 지원하게 설계했습니다.
  - 타임아웃은 **10초(10,000ms)**로 설정하고 패닉 시 재부팅되도록 기동했습니다.
  - 메인 스레드인 `loopTask`와 백그라운드 네트워크 처리를 담당하는 `NetworkTask`를 워치독에 각각 등록(`esp_task_wdt_add(NULL)`)하고 루프마다 `esp_task_wdt_reset()`을 통해 감시받게 했습니다.
  - **OTA 예외 처리:** 펌웨어 업데이트 모드(`STATE_OTA_UPDATING`) 진입 시, 메인 루프에서 `esp_task_wdt_delete(NULL)`를 통해 워치독 감시 대상에서 자발적으로 탈퇴합니다. OTA가 종료되거나 실패하여 다시 `STATE_IDLE`로 돌아올 경우 자동으로 재등록(`esp_task_wdt_add(NULL)`)되도록 구현했습니다.
- **[AutoOtaManager.cpp](../src/AutoOtaManager.cpp)**:
  - 펌웨어 바이너리를 내려받아 플래싱하기 직전(`httpUpdate.update()` 호출 전)에 백그라운드 태스크 자체를 워치독에서 탈퇴시켰습니다. 업데이트 도중 실패(`HTTP_UPDATE_FAILED` 또는 `HTTP_UPDATE_NO_UPDATES`)하여 대기 모드로 돌아갈 때만 워치독에 재등록하여 안심하고 플래싱할 수 있도록 이중 안전장치를 확보했습니다.

### 3.3. 선제적 새벽 자동 재부팅 (Proactive Reboot)
- **[PowerManager.cpp](../src/PowerManager.cpp)**:
  - 기기가 계속 켜져 있을 때 발생할 수 있는 잠재적인 누수 및 무선 모듈 성능 저하를 방지하기 위해 매일 **새벽 04:00 AM KST**에 시스템을 리셋합니다.
  - 동작 중 오작동을 차단하기 위해 상태가 `STATE_IDLE`이며 모터가 정지 상태(`!isMotorRunning()`)일 때만 실행합니다.
  - 재부팅 루프에 빠지지 않도록 NVS `Preferences`(네임스페이스 `"smartbox"`, 키 `"reboot_day"`)를 활용하여 현재 날짜인 `tm_yday`(day of year)를 체크 및 저장한 뒤 `ESP.restart()`를 기동합니다.

## 4. 검증 결과 (Verification Results)
- **플랫폼 빌드 결과:** 
  - `esp32-c6-devkitc-1` 환경에 대해 PlatformIO 컴파일 및 링킹을 수행하여 성공적으로 완료했습니다.
  - 최종 펌웨어 바이너리가 정상적으로 생성되었음을 검증했습니다.
- **Watchdog 및 OTA 예외 검증:**
  - OTA 모드 진입 시 시리얼 로그를 통해 `loopTask`와 `NetworkTask`가 정상 탈퇴함을 확인하고, OTA 도중 워치독 트리거에 의한 기기 강제 재설정이 발생하지 않음을 검증했습니다.
