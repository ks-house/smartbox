# 📊 스마트 자동 수거함 2차 설계 및 계획 수립 보고서

본 문서는 ESP32-C6 기반 스마트 자동 수거함 S/W 개발 계획 및 TDD 설계 수립 단계에서 오간 대화 내용과 아키텍처 설계 의사결정 사항을 상세하게 정리한 결과물 보고서입니다.

## 1. 대화 및 요구사항 진행 경과

1. **최초 아키텍처 설계 요청**:
   - 하드웨어 안전(12V 릴레이 H-브릿지 단락 방지)을 최우선으로 보장하고, 변경 및 확장이 용이한 소프트웨어 구조 요청.
   - 플랫폼에 무관하게 단위 테스트가 가능하도록 **TDD(RED-GREEN-REFACTOR) 방식**을 전제로 한 개발 설계서 작성을 요청받음.
2. **신규 요구사항 추가 및 적용성 분석**:
   - 코드를 즉시 변경하기 전, 무선 네트워크(Wi-Fi, SmartThings 연동) 및 비동기 웹 관리자 페이지(실시간 차트 모니터링, 감도 파라미터 Flash 저장 조정, 원격 강제 제어)의 아키텍처 적용 가능성을 검토해달라는 요청을 받음.
   - 분석 결과로서 FSM 루프 독립성을 위해 FreeRTOS 태스크/큐 분리 구조와 웹 비동기 소켓 핸들링 및 테스트 훼손을 막는 이벤트 콜백 패턴 설계안을 도출하여 검토함.
3. **최종 통합 및 문서 정리**:
   - 신규 기능들을 모두 개발 계획서 및 Task List에 정식 수립한 뒤, 임시 분석용 문서를 삭제하고 프로젝트 내 `docs/` 및 `reports/`, `prompts/` 폴더에서 문서를 정밀하게 통합 유지보수하기로 합의함.

## 2. 핵심 아키텍처 의존성 및 의사결정 기록

### ① HAL (Hardware Abstraction Layer) 도입
- **배경**: 임베디드 펌웨어 개발 특성상 실제 보드(ESP32-C6) 없이는 핵심 로직을 테스트하기 어려워 개발 주기가 길어집니다.
- **결정**: C++ 가상 클래스를 이용해 GPIO, 센서, 타이머를 추상화하는 `HardwareInterface` 계층을 정의합니다. PC 로컬 빌드 환경(`native`)에서 시간을 빠르게 감거나 센서 값을 임의로 주입하는 `MockHardware`를 사용하여, 비즈니스 로직(릴레이 인터락 및 갭 타이머, 과전류 비상 정지 등)을 100% 모의 검증하도록 결정했습니다.

### ② 옵저버 패턴을 통한 네트워크/웹 결산 분리
- **배경**: Wi-Fi 라이브러리 및 AsyncWebServer 등은 PC 환경(`native`)에서 빌드가 불가능하며 제어 모듈에 직접 들어가면 단위 테스트가 훼손됩니다.
- **결정**: `SmartBoxController` 내에는 무선 통신 관련 헤더를 전혀 포함시키지 않고, 오직 상태 천이 발생 시 외부 콜백을 호출하는 `StateChangeCallback` 구조만 노출합니다. 이를 통해 웹 서버 및 스마트싱스 전송 리포터는 제어 루프 바깥에서 이벤트를 구독하여 비동기 처리하도록 완벽히 디커플링했습니다.

### ③ FreeRTOS 선점형 멀티태스킹의 우선순위 설정
- **배경**: 스마트싱스 연동 Webhook(HTTP POST)은 네트워크 환경에 따라 지연이 심해 제어 루프를 블로킹할 수 있습니다.
- **결정**: ESP32-C6는 Single Core이므로 FreeRTOS 태스크의 우선순위를 조정합니다. 수거함의 생명선인 FSM 제어 루프와 센서 감시 태스크는 최고 우선순위로 두고, 대기 시간이 발생하는 HTTP API 웹훅 처리 태스크는 최하 우선순위로 할당하여 제어 실시간성을 철저히 지킵니다.
