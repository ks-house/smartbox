## 2. 원인 분석 (Root Cause Analysis)

### 결함 1: TLS setInsecure() 사용
HTTPS 연결 시 서버 인증서를 전혀 검증하지 않아 공격자가 동일 네트워크에서 가짜 서버를 올릴 경우 악성 펌웨어가 설치되거나 InfluxDB 토큰이 탈취될 수 있었습니다.

### 결함 2: JSON 이스케이프 미처리
Wi-Fi SSID에 `"` 또는 `\` 문자가 포함된 경우(예: `My "Home" WiFi`), `getScanResultsJson()`이 유효하지 않은 JSON을 반환합니다. OTA 실패 오류 메시지에도 특수문자가 포함될 수 있습니다.

### 결함 3: 개방형 SoftAP
`WifiManager::init()` 호출 시 빈 문자열(`""`)을 AP 비밀번호로 전달하여 ESP32 SoftAP가 비밀번호 없이 열렸습니다. 동일 물리 공간의 누구나 연결하여 HTTP Basic Auth 크리덴셜을 스니핑할 수 있었습니다.

## 3. 해결 설계 및 구현 (Solution Design & Implementation)

### 3.1. TLS Root CA 인증서 검증 적용

**[secrets.h.example](../include/secrets.h.example) / [secrets.h](../include/secrets.h)**

`SECRET_ROOT_CA_CERT` (Raw String Literal) 매크로 추가:
```cpp
#define SECRET_ROOT_CA_CERT R"(
-----BEGIN CERTIFICATE-----
... (ISRG Root X1 / Let's Encrypt Root CA PEM) ...
-----END CERTIFICATE-----
)"
```

**[AutoOtaManager.cpp](../src/AutoOtaManager.cpp)**
```diff
-client.setInsecure(); // Bypass ROOT CA validation
+client.setCACert(SECRET_ROOT_CA_CERT); // TLS Root CA verification

-client2.setInsecure();
+client2.setCACert(SECRET_ROOT_CA_CERT); // TLS Root CA verification
```
- version.json 요청과 firmware.bin 다운로드 양쪽 모두 인증서 검증 활성화

**[TelemetryManager.cpp](../src/TelemetryManager.cpp)**
```diff
-InfluxDBClient client(INFLUXDB_URL, INFLUXDB_ORG, INFLUXDB_BUCKET, INFLUXDB_TOKEN);
-client.setInsecure();
+InfluxDBClient client(INFLUXDB_URL, INFLUXDB_ORG, INFLUXDB_BUCKET, INFLUXDB_TOKEN, SECRET_ROOT_CA_CERT);
```
- InfluxDB 라이브러리의 5번째 생성자 인자로 CA 직접 주입

### 3.2. JSON 문자열 이스케이프 헬퍼 적용

**[WifiManager.cpp](../src/WifiManager.cpp) / [WebDashboard.cpp](../src/WebDashboard.cpp)**

각 파일의 익명 네임스페이스(anonymous namespace)에 동일한 헬퍼 추가 (내부 링키지, 외부 노출 없음):
```cpp
namespace {
    String escapeJson(const String& input) {
        String out;
        out.reserve(input.length() + 8);
        for (unsigned int i = 0; i < input.length(); i++) {
            char c = input.charAt(i);
            if      (c == '"')  { out += '\\'; out += '"'; }
            else if (c == '\\') { out += '\\'; out += '\\'; }
            else if (c == '\n') { out += '\\'; out += 'n'; }
            else if (c == '\r') { out += '\\'; out += 'r'; }
            else if (c == '\t') { out += '\\'; out += 't'; }
            else                { out += c; }
        }
        return out;
    }
}
```

적용 지점:
| 파일 | 필드 | 이유 |
|------|------|------|
| `WifiManager.cpp` | `WiFi.SSID(i)` | 공유기 이름에 `"`, `\` 포함 가능 |
| `WebDashboard.cpp` | `SECRET_FIRMWARE_URL` | URL 내 특수문자 방어 |
| `WebDashboard.cpp` | `AutoOtaManager::getOtaErrorMessage()` | 오류 메시지 내 특수문자 방어 |

### 3.3. SoftAP 비밀번호 강제화

**[secrets.h.example](../include/secrets.h.example) / [secrets.h](../include/secrets.h)**
```cpp
#define SECRET_AP_PASS "smartbox_admin"  // 기본 비밀번호 (최소 8자)
```

**[WifiManager.cpp](../src/WifiManager.cpp)**
```cpp
// Enforce minimum 8-character AP password
static const char* DEFAULT_AP_PASS = "smartbox_admin";
const char* effectiveApPass = apPass;
if (effectiveApPass == nullptr || strlen(effectiveApPass) < 8) {
    Serial.printf("[WIFI] WARNING: AP password too short. Forcing default.\n");
    effectiveApPass = DEFAULT_AP_PASS;
}
_apPass = effectiveApPass;
WiFi.softAP(apSsid, effectiveApPass);
```

**[main.cpp](../src/main.cpp)**
```diff
-WifiManager::init("SmartBox-WiFi", "", savedSsid.c_str(), savedPass.c_str());
+WifiManager::init("SmartBox-WiFi", SECRET_AP_PASS, savedSsid.c_str(), savedPass.c_str());
```

## 4. ROOT CA 인증서 획득 가이드

### 방법 A: openssl 명령어로 서버 인증서 직접 추출 (권장)
```bash
# NAS 서버의 인증서 체인 전체를 출력
openssl s_client -connect tworimpa.synology.me:4443 -showcerts </dev/null 2>/dev/null

# 출력된 인증서 블록 중 마지막 것(Root CA)이 SECRET_ROOT_CA_CERT에 입력할 값
```

### 방법 B: Let's Encrypt 공식 Root CA 다운로드
```bash
# ISRG Root X1 (Let's Encrypt Root CA) - 유효기간 2035년까지
curl -O https://letsencrypt.org/certs/isrgrootx1.pem
```

### 방법 C: Synology NAS 자체 서명(Self-Signed) 인증서 사용 시
1. Synology DSM > 보안 > 인증서 > 내보내기
2. 내보낸 `.pem` 파일의 내용을 `SECRET_ROOT_CA_CERT`에 입력

> [!IMPORTANT]
> NAS의 HTTPS 인증서가 Let's Encrypt 발급이라면 ISRG Root X1 또는 R3 중간 CA 중 하나가 Root CA입니다.
> 인증서 만료 시 NAS에서 갱신하면 `secrets.h`의 ROOT CA를 업데이트할 필요가 없습니다(Root CA는 수년간 유효).
